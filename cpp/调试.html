基础知识：警告和测试

术语和技巧

警告 编译器消息暗示可能有问题的运行时行为/微妙的陷阱 （见下文）
断言 用于比较和报告表达式的预期值和实际值的语句 （见下文）
测试 比较零件或整个程序的实际和预期行为 （见下文）
代码覆盖率 实际执行和/或测试了多少代码 冠状病毒 ，…
静态分析 通过分析源代码发现潜在的运行时问题，如未定义的行为 阿山 、 乌山 、…
动态分析 通过运行实际程序发现内存泄漏等潜在问题 瓦尔格林 ，…
调试 在运行时单步执行代码并检查内存中的值 （ 下一个 ）
剖析 找出每个函数/循环/代码块对总运行时间、内存消耗等的贡献程度……
微基准测试 测量单个函数或语句/调用块而不是整个程序运行的运行时间的小测试

记住：使用专用类型！

  • 限制输入参数值
  • 确保中间结果的有效性
  • 保证返回值的有效性

目标：编译器作为正确性检查器 -如果它编译，它应该是正确的

编译器警告

  • 编译器错误=程序不可编译
  • 编译器警告=程序可编译，但有一段有问题的代码可能会导致运行时错误

gcc/clang Options

Most Important

-Wall  Highly recommended. You should always use at least this. It doesn’t really enable all warnings, but rather the most important ones that don’t produce too much false positive noise.

-Wextra  Enable even more warnings than -Wall. Highly recommended.

-Wpedantic  Highly recommended. Issue all warnings demanded by strict ISO C++; reject compiler-specific extensions.

-Werror  Treat all warnings as errors ⇒ any warning will terminate compilation

Recommended Set (Production Level)

-Wall -Wextra -Wpedantic -Werror

-Wconversion -Wfloat-equal -Wformat-nonliteral -Wformat-security -Wformat-y2k -Wformat=2 -Wimport -Winvalid-pch -Wlogical-op -Wmissing-declarations -Wmissing-field-initializers -Wmissing-format-attribute -Wmissing-include-dirs -Wmissing-noreturn -Wnested-externs -Wpacked -Wpointer-arith -Wredundant-decls -Wshadow -Wstack-protector -Wstrict-null-sentinel -Wswitch-enum -Wundef -Wwrite-strings

High Performance / Low Memory / Security

Might be very noisy!

MS Visual Studio Options

/W1  1级：严重警告

/W2  级别 2：重要警告

/W3  级别 3：生产级别警告。你应该至少使用这个。也是较新的 Visual Studio 项目的默认值。

/W4  强烈推荐，特别是对于新项目。并没有真正启用所有警告，而是启用不会产生太多误报噪音的最重要的警告。

/Wall  启用比 4 级更多的警告；可能有点太吵了。

/WX  将所有警告视为错误⇒任何警告都将终止编译

断言

运行时断言

如果表达式产生假则中止程序

用例：
  • 在运行时检查预期值/条件
  • 验证先决条件（输入值）
  • 验证不变量（例如，中间状态/结果）
  • 验证后置条件（输出/返回值）

应在发布版本中停用运行时断言以避免任何性能影响。

逗号必须用括号保护

assert 是一个预处理器宏（稍后会详细介绍），否则逗号将被解释为宏参数分隔符：

可以添加自定义宏（没有标准方法）：

（去）激活——g++/clang

通过定义预处理器宏来停用断言NDEBUG，例如，使用编译器开关：g++ -DNDEBUG …

（去）激活 – MS Visual Studio

断言被显式激活，如果

  • _DEBUG定义了预处理器宏，例如，使用编译器开关/D_DEBUG
  • /MDd提供编译器开关

如果NDEBUG定义了预处理器宏，断言将被显式停用 ；在项目设置中或使用编译器开关/DNDEBUG

静态断言 C++11

static_assert(bool_constexpr, “message”);

static_assert(bool_constexpr); C++17

如果常量表达式产生 false则中止编译

测试

指南

使用断言

检查尚未由类型表达/保证的期望/假设：

  • 仅在运行时可用的预期值
  • 先决条件（输入值）
  • 不变量（例如，中间状态/结果）
  • 后置条件（输出/返回值）

应在发布版本中 停用 运行时断言以避免任何性能影响。

编写单元测试

一旦确定了函数或类型的基本用途和接口。

  • 更快的开发：不需要耗时的日志记录和调试会话
  • 更容易的性能调优：可以不断检查是否仍然正确
  • 文档：期望/假设写在代码中

使用测试框架

更方便且不易出错：预定义检查、设置工具、测试运行器等。

初学者/小型项目： doctest
  • 非常紧凑和自我记录的风格
  • 简单设置：只包含一个标题
  • 编译速度非常快
更大的项目： Catch2
  • 与 doctest 相同的基本理念（doctest 在 Catch 之后建模）
  • 用于使用不同值执行相同测试的值生成器
  • 使用计时器、平均等进行微基准测试。
  • 比 doctest 编译慢，设置稍微复杂

doctest – 简单的测试用例

不要使用cin/ cout/cerr直接！

直接使用全局 I/O 流会使函数或类型难以测试：

在函数中：通过引用传递流

类范围：存储流指针

但是：尝试编写独立于流或任何其他特定 I/O 方法的类型。
